# Project Architecture

This project follows a pragmatic Clean Architecture. Keep business rules separate from delivery mechanisms such as React, HTTP, databases, and external services.

Dependencies point inward:

```text
Web / Database / External services
              |
              v
       Application use cases
              |
              v
          Domain core
```

Inner layers must not depend on outer layers. Domain logic should not care whether it is called from a React page, an HTTP endpoint, a CLI, or a test.

## Source Layers

```text
src/
|-- core/    Domain models, shared types, and pure business rules
|-- server/  Application use cases, persistence, and server-side services
|-- web/     HTTP/API routes, pages, layouts, and router setup
`-- client/  Reusable browser UI, hooks, contexts, and client utilities
```

Use these dependency directions:

```text
web    -> client, server, core
server -> core
client -> core
core   -> no other project layer
```

Do not introduce dependencies that point in the opposite direction. If logic in `core` needs a browser, server, framework, database, or network dependency, keep the abstraction in `core` and implement it in an outer layer.

## Web Boundaries

Files under `src/web` are delivery adapters and application composition points. They may connect routes to client components and server use cases, but should keep business logic out of route handlers, loaders, and components.

For example:

```text
web/routes/(app)/_authed/diary/index.tsx
        |
        v
server/measurements/measurements.ts
        |
        v
core/measurements/*
        |
        v
core/types/*
```

Keep API handlers thin: validate or translate the HTTP request, call the relevant server use case, and translate the result into an HTTP response. Keep reusable React code in `src/client` rather than route modules.

`src/web/routeTree.gen.ts` is generated by TanStack Router. Do not edit it manually.

## Existing Boundary Exceptions

Some existing files in `src/core/lib` still depend on `client` or `server`. Treat these as migration candidates, not patterns for new code. When changing them, prefer moving environment-specific orchestration to the appropriate outer layer.
